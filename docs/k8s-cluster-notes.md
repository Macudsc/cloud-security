# Managed Kubernetes: замечание по Security Group

При создании node group было зафиксировано длительное состояние PROVISIONING. Виртуальные машины узлов были созданы, однако узлы некоторое время не переходили в состояние Ready. Причина была связана с тем, что кластер использовал Security Group без необходимых правил для доступа к Kubernetes API и внутреннего взаимодействия.

После добавления правил для TCP/443, внутреннего трафика в диапазоне подсетей стенда и исходящего трафика узлы успешно зарегистрировались в кластере и перешли в состояние Ready.

Данный эпизод подтверждает практическую значимость контроля сетевых правил и Security Groups как меры предотвращения ошибок конфигурации облачных ресурсов.
