### Дополнение по Kubernetes NetworkPolicy

При проверке защищенного namespace `secure` попытка создать тестовый pod без securityContext была заблокирована политикой Pod Security `restricted`. Это подтверждает, что namespace не допускает запуск рабочих нагрузок без базовых параметров безопасности.

Для проверки NetworkPolicy были созданы два тестовых pod с корректным securityContext: `denied-client` без разрешающей метки и `allowed-client` с меткой `role=allowed-client`. Подключение от `denied-client` к приложению было заблокировано, а подключение от `allowed-client` было разрешено. Это подтверждает работу политики `default deny` и точечного разрешения сетевого взаимодействия.

## 4. Kubernetes: hardening контейнерной инфраструктуры

| Проверка | Состояние «до» | Состояние «после» | Результат |
|---|---|---|---|
| Запуск контейнера | контейнер vulnerable запускается с root/privileged-настройками | контейнер secure запускается с `runAsNonRoot`, `runAsUser: 10001`, `allowPrivilegeEscalation: false` | снижены последствия компрометации контейнера |
| Capabilities | не ограничены | выполнен `drop: ALL` | уменьшена поверхность привилегий контейнера |
| Root filesystem | доступен для записи | включен `readOnlyRootFilesystem: true` | снижена возможность изменения файловой системы контейнера |
| ServiceAccount | используется поведение по умолчанию | создан отдельный `ServiceAccount`, отключен automount token | снижен риск использования токена pod |
| RBAC | минимальные роли не заданы | создана ограниченная `Role` и `RoleBinding` | реализован принцип наименьших привилегий |
| NetworkPolicy | отсутствует | настроены `default deny` и разрешение только для `role=allowed-client` | ограничено горизонтальное сетевое взаимодействие |
| Проверка denied-client | доступ отсутствует, timeout | соответствует ожидаемому результату | лишний сетевой доступ заблокирован |
| Проверка allowed-client | доступ разрешён, получен ответ `secure demo app` | соответствует ожидаемому результату | разрешён только явно заданный сетевой сценарий |

Вывод: эксперимент подтвердил, что применение securityContext, RBAC, отдельного ServiceAccount, отключения automount токена, resource limits и NetworkPolicy позволяет снизить риски небезопасной конфигурации Kubernetes и ограничить горизонтальное сетевое взаимодействие внутри кластера.

### Проверка readinessProbe

Для проверки работы `readinessProbe` был выполнен негативный тест: порт проверки готовности временно изменён с `8080` на `9999`. После этого Kubernetes создал новый pod, который находился в состоянии `Running`, но не перешёл в состояние `Ready`. В EndpointSlice сервиса `demo-app-secure` этот pod был отмечен как `ready: false`, тогда как старый pod с корректной проверкой оставался `ready: true`.

После возврата корректной конфигурации deployment успешно восстановился, а неготовый pod был удалён. Проверка подтвердила, что `readinessProbe` предотвращает направление трафика на pod, который не прошёл проверку готовности.
